# 跨 Agent 状态交接的三种模式——为什么不能让 Agent B 从聊天记录里"猜"数据

> **系列**：云平台智能客服系统技术深度解析
> **关键词**：状态传递、Agent Chain、上下文注入、消息历史、AgentState


---

## 问题的本质

回顾之前提到的 FinOps 场景：

> 用户：*"我上个月云费用涨了 30%，帮我分析原因，看看有没有优化空间。"*

BillingAgent 产出了结构化的账单数据，FinOpsAgent 需要基于这些数据做分析。传递的是什么？不是一句话，是精确的数字、百分比、产品分类。

**传递方式直接决定了 FinOpsAgent 的分析质量。**

---

## 三种传递模式对比

```
模式 A：消息历史传递           模式 B：State 字段传递          模式 C：显式上下文注入
┌──────────┐                 ┌──────────┐                 ┌──────────┐
│ Agent A  │                 │ Agent A  │                 │ Agent A  │
│  outputs │                 │  writes  │                 │  outputs │
│  → chat  │                 │  → state │                 │  → raw   │
└────┬─────┘                 └────┬─────┘                 └────┬─────┘
     │                            │                            │
     ▼                            ▼                            ▼
┌──────────┐                 ┌──────────┐                 ┌──────────┐
│ Agent B  │                 │ Agent B  │                 │ Context  │
│ 从对话中  │                 │ 从State  │                 │ Builder  │
│ "提取"   │                 │  读取    │                 │ 格式化   │
└──────────┘                 └──────────┘                 └────┬─────┘
                                                              │
                                   ┌──────────┐               │
                                   │ Agent B  │◀──────────────┘
                                   │ 读取格式化 │
                                   │ 上下文     │
                                   └──────────┘
```

---

## 模式 A：消息历史传递（❌ 不推荐）

BillingAgent 输出的一段自然语言：
```
"您上月的总费用为 ¥12,580.00，其中 ECS ¥6,200.00（49.3%，环比+22.5%），
RDS ¥3,400.00（27.0%，环比+15.8%）……"
```

FinOpsAgent 需要从中提取精确数值。LLM 可能：
- 看错数字（¥12,580 → ¥12,850）
- 漏算条目（"其他费用"被忽略）
- 混淆百分比（49.3% 误关联到 RDS）

**结论**：依赖 LLM 从自然语言中"提取"数据，在一个 Agent 内是概率问题，跨 Agent 传递后变成累积误差。不推荐。

---

## 模式 B：State 字段传递（✅ 基础方案）

在 `AgentState` 中定义专用字段，Agent A 写入结构化数据，Agent B 直接读取：

```python
class FinOpsChainState(TypedDict):
    messages: Annotated[List[BaseMessage], add_messages]
    chain: List[str]               # 待执行的 Agent 序列
    chain_index: int
    agent_outputs: dict            # ← 所有 Agent 的结构化产出
```

BillingAgent 写入：
```python
state["agent_outputs"]["billing"] = {
    "period": "2025-06",
    "total_cost": 12580.00,
    "mom_change_percent": 18.2,
    "details": [
        {"product": "ECS", "cost": 6200.00, "ratio": 49.3, "mom_change": 22.5},
        {"product": "RDS", "cost": 3400.00, "ratio": 27.0, "mom_change": 15.8},
    ]
}
```

FinOpsAgent 直接获取精确数据——无损失，无歧义。

---

## 模式 C：显式上下文注入（✅ 增强方案）

光有精确数据还不够。FinOpsAgent 需要的是"带有分析指引的上下文"——不仅要知道数字，还要知道"基于这些数字做什么"。

这就是 **ContextBuilder** 的用处（Day 06 详述）。

---

## 本项目的混合方案

```
AgentState.agent_outputs  ──→ 结构化数据（精确传递）
         +
ContextBuilder            ──→ 分析指引（任务描述）
         =
     高质量的状态交接
```

---

## 核心要点

1. **不要让 Agent B 从聊天记录里"猜"数据**——用结构化字段传递，精确且类型安全；
2. **State 字段是基础设施**——`agent_outputs` 字典承载所有结构化中间产物；
3. **只传数据还不够**——下游 Agent 需要知道"拿这些数据做什么"。
