> **系列**：云平台智能客服系统技术深度解析

## 回顾三层容错结构

```
用户问题
   │
   ▼
Layer 1: LLM Text2Cypher ──── 85% 成功率，最智能但不可靠
   │ (失败)
   ▼
Layer 2: 预定义模板匹配 ──── 100% 可靠，但有覆盖盲区
   │ (未匹配)
   ▼
Layer 3: 关键词全文检索 ──── 永远有结果，精度最低但保证不死机
```

Day 09 覆盖了 Layer 1。今天深入 Layer 2 和 Layer 3。

## 一、Layer 2：预定义查询模板

模板是**经过人工验证、100% 可靠的 Cypher 查询**。每个模板包含正则匹配模式和参数化查询：

```python
CYPHER_TEMPLATES = {
    "instance_spec": {
        "name": "查询实例规格",
        "patterns": [
            r"(?P<instance>\w+\.?\w*)\s*(?:的|有什么|支持|多大|多少)\s*(?:规格|CPU|内存|网卡|带宽|配置)",
        ],
        "cypher": """
            MATCH (i:InstanceType)
            WHERE toLower(i.name) = toLower($instance)
            RETURN i.name, i.cpu, i.memory,
                   i.network_bandwidth, i.price_per_hour
        """,
        "params": ["instance"]
    },

    "region_availability": {
        "name": "查询地域可用实例",
        "patterns": [
            r"(?P<region>\S+)\s*(?:地域|区域|可用区)\s*(?:支持|有|可用)\s*(?:哪些|什么)",
        ],
        "cypher": """
            MATCH (i:InstanceType)-[:AVAILABLE_IN]->(r:Region)
            WHERE toLower(r.name) CONTAINS toLower($region)
               OR toLower(r.code) = toLower($region)
            RETURN i.name, i.cpu, i.memory, i.price_per_hour
            ORDER BY i.price_per_hour
        """,
        "params": ["region"]
    },

    "product_comparison": {
        "name": "产品对比",
        "patterns": [
            r"(?:对比|比较|区别|差异)\s*(?P<product_a>\S+)\s*(?:和|与|跟)\s*(?P<product_b>\S+)",
        ],
        "cypher": """
            MATCH (a:InstanceType) WHERE toLower(a.name) = toLower($product_a)
            MATCH (b:InstanceType) WHERE toLower(b.name) = toLower($product_b)
            RETURN a.name, a.cpu, a.memory, a.price_per_hour,
                   b.name, b.cpu, b.memory, b.price_per_hour
        """,
        "params": ["product_a", "product_b"]
    },
}
```

**模板的设计原则**：
- 每个模板覆盖一类高频问题；
- 正则匹配从问题中提取参数（实例名、地域名等）；
- Cypher 使用参数化查询，防止注入。

## 二、Layer 3：关键词全文检索——最后的防线

当 LLM 生成的 Cypher 失败、且模板无法匹配时，进入最终兜底层。思路是：从问题中提取实体，在 Neo4j 中做模糊全文检索，把"候选集"交给 LLM 做最终筛选。

```python
class KeywordFallbackSearcher:
    ENTITY_PATTERNS = {
        "instance": [
            r'\b([A-Za-z]\d+\.?\w*)\b',       # C7.4xlarge
            r'\b([a-z]+\d+\.?\w*)\b',         # ecs.c7.xlarge
        ],
        "region": [
            r'(华东[12]|华南[12]|华北[123])',
            r'(北京|上海|杭州|深圳|广州|成都|香港)',
        ],
        "series": [r'([CGRDEFHTM]\d+)\s*系列?'],  # C7系列
    }

    def extract_entities(self, question: str) -> dict:
        entities = {}
        for entity_type, patterns in self.ENTITY_PATTERNS.items():
            found = []
            for pattern in patterns:
                found.extend(re.findall(pattern, question))
            if found:
                entities[entity_type] = list(set(found))
        return entities
```

然后基于提取的实体构建模糊查询：
```cypher
MATCH (i:InstanceType)
WHERE toLower(i.name) CONTAINS "c7"
OPTIONAL MATCH (i)-[:AVAILABLE_IN]->(r:Region)
WHERE toLower(r.name) CONTAINS "杭州"
RETURN i.name, i.cpu, i.memory, i.price_per_hour
LIMIT 20
```

关键词检索返回的是"候选集"——可能包含相关和不相关的结果。需要 LLM 做最后一层过滤和回答生成。

## 三、监控驱动的持续优化

```python
class GraphQueryMetrics:
    def report(self) -> dict:
        total = self.layer1_attempts
        return {
            "Layer1成功率": f"{self.layer1_successes / max(total, 1) * 100:.1f}%",
            "Layer2触发率": f"{self.layer2_attempts / max(total, 1) * 100:.1f}%",
            "Layer3触发率": f"{self.layer3_attempts / max(total, 1) * 100:.1f}%",
            "完全失败率": f"{self.total_failures / max(total, 1) * 100:.1f}%",
        }
```

**理想指标**：
- Layer 1 成功率 > 85%
- Layer 3 触发率 < 5%（过多触发说明上层需要优化）
- 完全失败率 < 1%

---

## 核心要点

1. **模板是确定性的保障**——20 个精心设计的模板覆盖 80% 的高频查询场景；
2. **关键词兜底保证系统"永不死机"**——即使 LLM 完全瞎编，也能从实体提取 + 模糊检索中得到一些结果；
3. **监控 Layer 3 的触发频率**——这是衡量上游 prompt 质量的最直接信号。
