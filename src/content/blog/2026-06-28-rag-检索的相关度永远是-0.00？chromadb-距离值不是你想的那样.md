# RAG 检索的相关度永远是 0.00？ChromaDB 距离值不是你想的那样

v0.1 版本完成后，我兴冲冲地测了下 RAG 问答功能。语义检索能召回东西——这很好。但前端显示的"相关度分数"永远是 `0.00`。

这就很尴尬。笔记命中了，答案生成了，但用户看到的是 **"相关度: 0.00"**，像在说"我们随便找了几条不相关的东西"。

## 排查

ChromaDB 的 `query` 返回的数据结构长这样：

```python
{
    'ids': [['chunk_abc123', 'chunk_def456']],
    'documents': [['这是笔记内容...', '这是另一个笔记...']],
    'distances': [[0.23, 0.87]],  # ← 这里
    'metadatas': [[{...}, {...}]],
}
```

看到 `distances` 了吗？**ChromaDB 默认用的是 L2 距离（欧氏距离）。**

L2 距离的含义是：**越小越相关，0 = 完全匹配。**

但前端代码是这么显示的：

```python
# 原来的代码
"score": metadata.get("distance", 0)  # 直接把距离当成分数
```

L2 距离 0.23 意味着"很相关"，但显示成 0.23 的分数，用户看起来就是"基本不相关"。

而 L2 距离 0.87 的 chunk 其实不太相关，但显示 0.87，用户会觉得"这个更靠谱"。

**大小关系反了。** 而且数量级也不对——用户期望看到的是 0 到 1 的"相关度百分比"。

## 修复

写一个转换函数：

```python
def _distance_to_relevance(distance: float) -> float:
    """将 ChromaDB 距离值转换为相关度分数 (0-1，越高越相关)"""
    return round(1.0 / (1.0 + abs(distance)), 4)
```

这个公式的效果：

| L2 距离 | 相关度 | 含义 |
|---------|--------|------|
| 0.0 | 1.0000 | 完美匹配 |
| 0.23 | 0.8130 | 很相关 |
| 0.5 | 0.6667 | 一般相关 |
| 0.87 | 0.5348 | 不太相关 |
| 2.0 | 0.3333 | 基本不相关 |

然后在 VectorStore 的 `search()` 方法里调用它：

```python
# vector_store.py 第 111 行
"score": _distance_to_relevance(
    results["distances"][0][i]
) if results["distances"] else 0.0,
```

## 另一个相关的问题

前端和检索 Agent 之间还有一个字段名不一致的问题。VectorStore 返回 `score`，但 RetrievalAgent 传给 LLM 的 prompt 里用的是 `relevance_score`，前端也是读的 `relevance_score`。

最后我在 RetrievalAgent 里加了个 `_enrich_sources()` 函数，LLM 返回后自动把检索到的 score 按 title 匹配注入到 sources 里。这样不管 LLM 输出什么格式，前端拿到的都是带真实相关度的数据。

修复后效果很明显：

```
旧版：来源 "软件测试基础" (相关度: 0.00)
新版：来源 "软件测试基础" (相关度: 0.87)
```

用户终于不用怀疑自己搜到的是不是随机内容了。

## 教训

1. **向量数据库的 "distance" 不是 "similarity"。** 用之前搞清楚是 L2、Cosine 还是 Inner Product，每种的含义不一样。
2. **数据流向的每一环都要检查字段名。** VectorStore → RetrievalAgent → API Response → 前端，四个环节任意一个字段名不一致，用户看到的都是 0。
3. **0.00 是典型的"静默失败"。** 如果代码不报错，但显示值永远不对，大概率是数据转换环节出了问题。
