# 缓存安全与预热——用户 A 的账单数据绝不能进入公共缓存

> **系列**：云平台智能客服系统技术深度解析
> **关键词**：缓存安全、作用域隔离、用户数据、缓存预热、缓存淘汰


---

## 一、缓存的安全陷阱

假设用户 A 问了"我有哪些 ECS 实例"，系统生成了答案，存入了缓存。5 分钟后用户 B 问了同样的问题——他应该看到**自己的**实例，而非用户 A 的。

**如果这条缓存是 public 作用域**，用户 B 就会看到用户 A 的实例列表。这是一起严重的数据泄露事故。

---

## 二、作用域隔离机制

```python
class CacheScopeDecider:
    """自动判断 Q&A 的安全作用域"""

    # 绝对不能进 public 缓存的信号
    USER_SPECIFIC_PATTERNS = [
        r'(?:我的?|本人|个人|账号|账户)',
        r'(?:上月|本月|本季度|今年)\w*(?:账单|费用|消费|订单)',
        r'(?:实例|资源|服务器)\w*(?:列表|清单|数量)',
        r'(?:偏好|喜欢|习惯)',
    ]

    # 可以进 public 的知识类别
    PUBLIC_SAFE_CATEGORIES = [
        "product_spec",       # 产品规格——人人相同
        "product_concept",    # 概念解释——人人相同
        "pricing_info",       # 公开定价——人人相同
        "how_to_guide",       # 操作指南——人人相同
        "faq",
    ]

    def decide_scope(self, question: str, category: str) -> str:
        # 规则1：公开知识类别 → public
        if category in self.PUBLIC_SAFE_CATEGORIES:
            return "public"

        # 规则2：问题包含用户特定标记 → user
        for pattern in self.USER_SPECIFIC_PATTERNS:
            if re.search(pattern, question):
                return "user"

        return "public"
```

**两条规则解决安全问题**：
- 产品规格类的知识（"C7.4xlarge 多少钱"）→ public，人人可用
- 含"我的"、"账单"、"实例列表"等标记 → user，仅自己可用

## 三、缓存预热——不让首批用户"替系统热身"

系统重启后缓存为空，前 N 个用户承受完整延迟。预热脚本解决这个问题：

```python
class CacheWarmer:
    # 从历史日志提取的高频问题 Top-N
    HOT_QUESTIONS = [
        ("C7.4xlarge 的详细配置是什么？", "product_spec", "public"),
        ("什么是弹性伸缩？", "product_concept", "public"),
        ("如何开通云服务器 ECS？", "how_to_guide", "public"),
        ("RDS 和自建数据库有什么区别？", "product_concept", "public"),
        ("SLB 的几种类型有什么区别？", "product_concept", "public"),
        # ... 更多高频问题
    ]

    async def warm_up(self, exact_cache, semantic_cache, agent_executor):
        semaphore = asyncio.Semaphore(5)  # 控制并发，不打死 LLM API

        async def warm_one(question, category, scope):
            async with semaphore:
                result = await agent_executor.ainvoke(question)
                # 同时写入 L1 和 L2
                await exact_cache.set("system", question, result, scope)
                await semantic_cache.store(question, result, "system", scope, category)

        tasks = [warm_one(q, cat, scope) for q, cat, scope in self.HOT_QUESTIONS]
        await asyncio.gather(*tasks)
```

预热脚本在系统启动后异步运行，首批用户访问时缓存已经"热"了。

## 四、缓存淘汰——自动清理

不同过期策略的组合拳：

```python
class CacheEvictionManager:
    async def periodic_cleanup(self):
        while True:
            # 1. TTL 过期——Milvus 中 7 天以上的向量缓存
            cutoff = int(time.time()) - 7 * 24 * 3600
            semantic_cache.delete(f"created_at < {cutoff}")

            # 2. 低命中率淘汰——30 天内命中 < 2 次的"垃圾缓存"
            low_hit_cutoff = int(time.time()) - 30 * 24 * 3600
            semantic_cache.delete(
                f"created_at < {low_hit_cutoff} AND hit_count < 2"
            )

            # 3. Redis TTL 由 Redis 自身管理
            await asyncio.sleep(6 * 3600)  # 每 6 小时执行一次
```

**TTL 淘汰**清理过期数据，**低频淘汰**清理"僵尸缓存"——那些被存储但几乎从未被命中的 Q&A。

---

## 本期核心要点

1. **公共缓存 ≠ 无限制共享**——作用域隔离是数据安全的基础保障；
2. **预热脚本让缓存"开箱即热"**——首批用户体验不应被冷启动影响；
3. **TTL + 低频淘汰双管齐下**——过期缓存自动清理，无需人工维护。
