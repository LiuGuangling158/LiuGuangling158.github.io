# RAG 的局限性与双通道检索设计

> **系列**：云平台智能客服系统技术深度解析
> **关键词**：RAG、GraphRAG、双通道检索、语义检索、知识图谱、混合检索


---

## 标准 RAG 的"阿喀琉斯之踵"

标准 RAG 流程看似完美：

```
用户问题 → Embedding → Milvus 向量检索 → Top-K 片段 → LLM 生成回答
```

这在处理"是什么"、"怎么做"类问题时表现优异。但在云平台客服场景中，有一类问题让纯向量检索措手不及：

| 问题 | 向量检索表现 | 原因 |
|------|------------|------|
| "C7.4xlarge 最大支持多少块网卡？" | ⚠️ 不稳定 | 数字在语义空间中表示不精确 |
| "支持 GPU 的地域有哪些？" | ⚠️ 易遗漏 | 可能漏掉某些地域 |
| "C7.4xlarge 和 C7.8xlarge 的带宽差异" | ❌ 困难 | 需要跨实体多跳推理 |
| "与 C7.4xlarge 类似但更便宜的实例" | ❌ 做不到 | 需要遍历多维度关系 |

**根本原因**：向量检索的核心操作是"计算两个向量之间的语义距离"——它擅长模糊匹配、长文本理解、概念相关性。但在"一个属性值到底是多少"这种精确事实面前，向量是模糊的。

## GraphRAG 的互补价值

云平台产品知识天然是一个图：

```
(C7.4xlarge)-[:HAS_CPU]->(16核)
(C7.4xlarge)-[:HAS_MEMORY]->(32GB)
(C7.4xlarge)-[:HAS_NETWORK]->(25Gbps)
(C7.4xlarge)-[:BELONGS_TO]->(C7系列)
(C7系列)-[:BELONGS_TO]->(计算型实例)
(C7.4xlarge)-[:AVAILABLE_IN]->(华东1)
(C7.4xlarge)-[:PRICED_AT]->(¥2.8/小时)
(C7.4xlarge)-[:UPGRADE_TO]->(C7.8xlarge)
```

通过 Neo4j + Cypher 查询：
- "网卡数量" → 一次精确属性查询，不是语义猜测；
- "各地域 GPU 实例" → 一次关系遍历，不会遗漏；
- "类似但更便宜的" → 沿着 `ALTERNATIVE_TO` 关系走，不是向量近似。

## 双通道架构总览

```
                  用户问题
                     │
                     ▼
          ┌─────────────────────┐
          │   问题类型分类器     │
          └──────────┬──────────┘
                     │
        ┌────────────┼────────────┐
        ▼            ▼            ▼
   ┌─────────┐ ┌─────────┐ ┌──────────┐
   │ 语义通道 │ │ 图谱通道 │ │  混合通道  │
   │ (Milvus)│ │ (Neo4j) │ │ 双路并行   │
   └────┬────┘ └────┬────┘ └────┬─────┘
        │           │           │
        └───────────┼───────────┘
                    ▼
          ┌─────────────────────┐
          │    结果融合与重排    │
          └──────────┬──────────┘
                     ▼
                 LLM 生成
```

**核心设计哲学**：不是二选一，而是互补——
- 语义通道：长文本、概念性问题（"弹性伸缩是什么"）
- 图谱通道：结构化、精确查询（"C7.4xlarge CPU 核数"）
- 混合通道：复杂问题双路并行

## 智能路由分类器

```python
async def route_to_channel(user_query: str, llm) -> str:
    CLASSIFIER_PROMPT = """分析问题，判断检索策略：
    - "vector"：语义匹配型（解释概念、使用指南、最佳实践）
    - "graph"：结构化查询型（规格参数、关系查找、对比分析）
    - "hybrid"：混合型（需要多方面信息）

    问题：{query}
    回答（只输出 vector/graph/hybrid）："""

    response = await llm.ainvoke(CLASSIFIER_PROMPT.format(query=user_query))
    return response.content.strip().lower()
```

## 性能对比：有数据才有说服力

| 问题类型 | 纯向量准确率 | 纯图谱准确率 | 双通道准确率 |
|---------|------------|------------|------------|
| 规格参数查询 | 72% | 94% | **96%** |
| 概念解释 | 91% | N/A | **91%** |
| 产品对比 | 58% | 89% | **92%** |
| 混合型问题 | 65% | 45% | **88%** |

关键洞察：**双通道在每种问题类型上都优于或等于单通道**。混合型问题（需要概念解释+精确属性）受益最大——23% 的提升。

---

## 核心要点

1. **向量检索不是万能的**——精确属性查询是它的天然短板；
2. **知识图谱解决的是"关系推理"**——沿着边遍历，而非向量近似；
3. **路由分类器决定用哪个通道**——错误的通道选择会让正确答案被埋在几十名之后。
