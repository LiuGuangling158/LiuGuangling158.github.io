# LangChain 学习记录（Day 4）

## 今天学了什么

今天开始正式接触 RAG。

之前刷 B 站和看技术文章的时候，经常能看到：

* AI 知识库
* 企业知识库问答
* AI 客服
* AI 助手

这些项目基本都会提到 RAG。

刚开始我一直以为这是某种训练模型的方法。

后来才发现完全不是。

RAG 的核心思路其实很简单：

```text
用户提问
↓
先找资料
↓
把资料发给大模型
↓
生成答案
```

本质上就是让 AI 学会查资料。

---

## 第一个疑问

学习之前我有个问题。

既然 GPT 这么强。

为什么还要做知识库？

后来试着问了一些关于自己项目的问题。

比如：

```text
我的个人博客使用什么技术栈？
```

模型给出的回答有些正确。

有些完全是编的。

这时候才意识到：

大模型并不知道我的项目。

它只能根据训练数据和上下文去猜。

如果想让它准确回答。

就必须把相关资料给它。

于是知识库的作用就出现了。

---

## Loader 比想象中重要

今天最开始学的是 Loader。

刚看到这个模块的时候感觉有点无聊。

因为它做的事情看起来特别简单：

```text
读取文件
```

例如：

```python
TextLoader

PyPDFLoader

CSVLoader
```

感觉自己写几行 Python 也能实现。

后来继续往下学才发现。

LangChain 其实是在统一数据入口。

因为真实项目里的数据来源太杂了。

可能有：

* PDF
* Word
* Markdown
* 网页
* 数据库

如果每种格式都单独处理。

后面的代码会越来越乱。

Loader 的作用就是先把这些内容转换成统一的 Document 对象。

这点有点像后端开发里的 DTO。

---

## 第一次接触 Document

读取文档后发现返回的是：

```python
Document
```

里面主要有两部分内容：

```python
page_content

metadata
```

最开始没太在意。

后来发现 metadata 其实挺有用。

例如：

```text
文件名

页码

来源路径
```

这些信息后面检索的时候都可能用到。

感觉 LangChain 在设计上还是挺偏工程化的。

很多东西一开始觉得多余。

后面都会发现有存在的理由。

---

## 文档为什么要切分

接下来学的是 Text Splitter。

刚开始看到这一步的时候其实有点不理解。

因为：

```text
文件都读出来了

为什么还要拆开？
```

后来想了一下。

如果是一篇博客：

```text
3000字
```

问题不大。

但如果是一整本书：

```text
几十万字
```

显然不可能全部发给模型。

于是就需要：

```text
大文档
↓
拆成小块
```

这个过程就是 Chunk。

---

## 今天踩的第一个坑

刚开始测试的时候。

我把 Chunk Size 设置得特别小。

结果检索效果很差。

举个例子：

原文：

```text
张三毕业于北京大学，
毕业后进入腾讯工作。
```

切分后变成：

```text
Chunk1:
张三毕业于北京大学

Chunk2:
毕业后进入腾讯工作
```

然后提问：

```text
张三毕业后去了哪里？
```

结果模型回答得乱七八糟。

因为相关信息被拆开了。

后来才明白。

Chunk 并不是切得越细越好。

切分策略本身也是 RAG 很重要的一部分。

---

## Embedding 终于没那么抽象了

今天花时间最多的其实是 Embedding。

这是我之前最看不懂的部分。

网上很多文章一上来就是：

```text
高维向量空间

余弦相似度

向量距离
```

看得头疼。

后来换了个思路理解。

假设有两个词：

```text
Java

Spring Boot
```

它们经常一起出现。

语义上也比较接近。

Embedding 模型会把它们映射到比较接近的位置。

而：

```text
Java

篮球
```

距离就会远很多。

虽然最后生成的是一堆数字。

但本质上是在给文本建立坐标。

这样计算机就能计算：

```text
谁和谁更像
```

想到这里突然觉得没那么神秘了。

---

## 第一次理解向量数据库

之前一直听别人说：

```text
向量数据库
```

但不知道到底和普通数据库有什么区别。

今天搭配 Embedding 学习之后终于有点感觉。

传统数据库更像：

```sql
查找包含 Java 的文章
```

而向量检索更像：

```text
查找和后端开发相关的文章
```

即使文章里没有出现“后端开发”这个词。

只要语义接近。

也有机会被找到。

这种感觉和搜索引擎已经完全不一样了。

---

## 用 FAISS 建立第一个索引

今天用的是 FAISS。

过程大概是：

```text
Markdown文件
↓
Loader
↓
Text Splitter
↓
Embedding
↓
FAISS
```

整个流程跑通之后。

终于可以根据问题检索自己的文档了。

虽然只是一个很简单的 Demo。

但看到检索结果正确出现的时候还是挺有成就感的。

因为前几天学的东西终于串起来了。

---

## 一个新的认识

今天学习过程中最大的感受是：

以前总觉得 AI 的核心是模型。

现在越来越觉得：

```text
模型能力只是一部分
```

很多企业项目真正投入精力的地方可能是：

* 数据处理
* 检索优化
* 知识库维护
* 系统架构

模型反而是其中最稳定的一环。

这和软件工程其实很像。

用户看到的是最终产品。

但真正复杂的往往是背后的工程实现。

---

## 目前还没搞懂的地方

今天还有几个问题没想明白：

### 检索结果为什么有时候不准

同一个问题。

有时候能找到正确内容。

有时候却会漏掉。

感觉和 Chunk 的设计有关系。

后面准备继续研究。

### Embedding 模型怎么选

目前只是用默认模型测试。

但不同模型之间具体差异还不清楚。

后面准备找时间对比一下。

### 企业知识库规模有多大

现在测试的都是几篇 Markdown。

如果换成：

```text
10万份文档
```

甚至：

```text
100万份文档
```

检索逻辑会不会发生变化。

这个问题暂时还没有答案。
