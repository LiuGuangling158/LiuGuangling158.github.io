# 工具定义与 ReAct 调优实战——好工具描述是成功的一半

> **系列**：云平台智能客服系统技术深度解析
> **关键词**：工具定义、Tool Description、ReAct 调优、温度控制、重复调用检测


---

## 一、工具描述就是你的 API 文档

很多开发者低估了工具描述的重要性。LLM 选择工具的依据**只有**工具名称和描述——它看不到源码，不知道你背后是查 MySQL 还是调 API。

来看看好和差的区别：

```python
# ❌ 差的工具定义——LLM 不知道怎么用
@tool
def query_db(sql: str):
    """查询数据库"""
    pass

# ✅ 好的工具定义——LLM 看了就知道何时调用
@tool
def query_product_spec(instance_type: str, attributes: list[str]) -> dict:
    """查询云服务器实例的详细规格参数。

    适用场景：
    - 用户询问某款实例的 CPU、内存、网络、存储等参数
    - 对比多款实例的规格差异
    - 确认某地域是否提供特定实例类型

    参数说明：
    - instance_type: 实例规格代码，如 "C7.4xlarge"（大小写不敏感）
    - attributes: 需要查询的属性列表，可选值：
      ["cpu", "memory", "network_bandwidth", "pps", "storage", "gpu"]

    返回：包含请求属性的字典，缺失属性值为 None
    """
    pass
```

**好工具描述的三个要素**：
1. **适用场景** → 帮 LLM 判断"我该不该用这个工具"；
2. **参数示例** → 减少 LLM 传参格式错误；
3. **返回格式** → 让 LLM 知道如何解读结果。

## 二、工具数量：少即是多

产品 Agent 只挂载了三个工具——不是技术限制，是精心设计的结果：

```python
ProductAgent.tools = [
    MilvusSearchTool(),       # 语义检索 → 模糊问题、长文本理解
    Neo4jQueryTool(),         # 图谱查询 → 精确属性、关系推理
    KeywordFallbackTool(),    # 降级兜底 → 保障问答不出空白
]
```

**为什么三个刚刚好？**

- 2 个太少 → 覆盖面不足，某些问题无法回答；
- 3 个刚好 → 覆盖"语义→精确→兜底"完整链路；
- 5+ 个太多 → LLM 选择困难，prompt 膨胀，token 成本上升。

## 三、不同 Agent 不同温度

```python
AGENT_TEMPERATURES = {
    "billing":     0.0,   # 金额绝对不能有偏差
    "product":     0.1,   # 产品信息需要精确
    "promotion":   0.2,   # 促销规则基本确定
    "finops":      0.2,   # 成本分析需要准确
    "recommendation": 0.4, # 推荐需要一定创造性
}
```

Billing Agent 用 `temperature=0.0`——金额查询没有"创造性"的余地，差一分钱都是错误。而 Recommendation Agent 用 `temperature=0.4`——推荐方案需要一定多样性，太死板反而不好。

## 四、ReAct 常见问题与修复

### 问题1：LLM 拒绝调用工具

**现象**：Agent 用训练知识回答而非查询数据库。

**修复**——System Prompt 中明确规则：
```
关键规则：
1. 必须先通过工具查询数据，严禁依赖训练数据回答
2. 如果查询结果与训练知识冲突，以查询结果为准
3. 每次回答必须引用具体数据来源
```

### 问题2：循环调用同一工具

**现象**：Agent 反复用略有不同的参数查询同一件事。

**修复**——重复调用检测：
```python
class DeduplicatingToolNode(ToolNode):
    def __init__(self, tools):
        super().__init__(tools)
        self._call_history = []

    def _invoke(self, state):
        last_message = state["messages"][-1]
        for tc in last_message.tool_calls:
            # 检查是否与最近 3 次调用重复
            if self._is_duplicate(tc["name"], tc["args"]):
                return {"messages": [ToolMessage(
                    content="此查询与前次高度重复，请基于已有结果回答。",
                    tool_call_id=tc["id"]
                )]}
        return super()._invoke(state)
```

### 问题3：过早给出结论

**修复**——在 System Prompt 加入自检步骤：
```
在生成最终回答之前，执行以下自检：
1. 我是否获取了用户询问的所有信息？
2. 是否有任何数据点需要交叉验证？
3. 如果信息不完整，我应该再调用哪个工具？
```

## 核心要点

1. **工具描述决定 LLM 的工具使用质量**——投入时间写清楚"适用场景+参数示例+返回格式"；
2. **每个 Agent 的工具控制在 3-5 个**——多了 LLM 选择困难，少了覆盖不足；
3. **不同场景不同配置**——Billing 用 0 温度，Recommendation 用 0.4，没有一刀切的最佳值。
