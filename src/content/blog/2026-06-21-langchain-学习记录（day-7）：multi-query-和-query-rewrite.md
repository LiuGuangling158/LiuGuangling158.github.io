# LangChain 学习记录（Day 7）

## 今天在解决一个很烦的问题

昨天 Rerank 加上之后，检索结果稳定了一点。

但今天测试的时候又遇到一个更隐蔽的情况：

```
同一个意思，不同问法，结果完全不一样
```

## 一个例子

同一份文档，我问：

```
后端开发技术栈有哪些？
```

能正常搜到。

换一种说法：

```
做服务端开发要学什么？
```

结果就开始不稳定，有时候直接搜不到。


## 问题在哪

不是模型问题，也不是 FAISS 问题。

更像是：

```
人话 ≠ 向量能完全对上
```

它只能做“相似”，但不会理解“这是同一件事”。


# Multi-Query Retrieval

今天先试了 Multi-Query。

思路很简单：

```
一个问题 → 换几种问法一起搜
```

比如：

原问题：

```
服务端开发需要学什么？
```

拆成：

```
后端开发技术栈
服务端开发语言
后端学习路线
```

然后分别去检索，再合并结果。



## 体感

改完之后的变化：

* 之前：有时候搜不到
* 现在：基本都能捞出来

但问题也很明显：

* 搜的内容变多了
* 有些结果是重复的



# Query Rewrite

后来又看了 Query Rewrite。

这个更直接：

```
把用户的问题改成“更像搜索关键词”的形式
```

比如：

用户说：

```
我做这个项目需要学什么？
```

改写后可能变成：

```
RAG项目开发技术栈
```

或者更直白一点：

```
LangChain RAG需要哪些技术
```



## 和 Multi-Query 的区别

我自己的理解是：

```
Multi-Query：多问几种方式
Rewrite：换一种更标准的问法
```


## 一个实际问题

一开始我做 Multi-Query 的时候，写得很机械：

```
同义词替换
```

结果效果很差。

后来才发现问题：

```
不是换词，是换说法
```

比如：

* “后端开发”
* “服务端开发”
* “Web后端”

这些不是简单同义词，是不同表达方式。



# 今天的完整链路

现在的流程变成这样：

```
用户问题
↓
Query Rewrite
↓
Multi-Query
↓
Embedding 检索
↓
TopK
↓
Rerank
↓
LLM回答
```


# 今天踩的坑

## 1. query越多不一定越好

一开始我直接生成 10 个 query。

结果：

* 噪音变多
* 有些完全跑偏

后来缩到 3～5 个比较稳定。



## 2. rewrite容易“过度优化”

有时候 rewrite 会变成：

```
自己加理解
```

反而偏离原问题。



## 3. 成本开始明显增加

多 query + rerank 后：

* 请求变多
* latency变长

已经开始有点“工程问题”的感觉了。



# 今天的一个感受

以前觉得 RAG 是：

```
向量搜索 + 大模型
```

现在感觉更像：

```
一堆“让搜索更靠谱的补丁”
```



# 一个简单总结

今天做的事情可以一句话概括：

```
让系统不要“只听用户怎么说”，而是“多想几种问法”
```
