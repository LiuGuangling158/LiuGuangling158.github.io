# asyncio.create_task() 在同步方法里跑路，数据删不掉

Python 的 async/await 我一直是"能用就行"的水平——在 FastAPI handler 里加个 `async` 加个 `await`，能用就行，没深想过。

直到 debug 一个删除笔记的 Bug，发现 ChromaDB 里的向量数据怎么删都删不掉。

## 问题

`note_service.py` 里删除笔记的方法是同步的：

```python
# note_service.py (旧)
def delete_note(self, note_id: str) -> bool:      # ← 同步方法
    session = db_manager.get_session()
    try:
        note = session.query(Note).filter(Note.id == note_id).first()
        if not note:
            return False

        # 删 ChromaDB 的向量数据
        if note.embedding_id:
            try:
                import asyncio
                asyncio.create_task(
                    vector_store.delete_by_note(note_id)   # ← 这是 async 方法
                )
            except Exception as e:
                print(f"[WARN] ChromaDB 删除失败: {e}")

        session.delete(note)
        session.commit()
        return True
```

逻辑上看没问题：先从 SQLite 里删笔记，同时异步删 ChromaDB 里的向量块。但实际上 ChromaDB 的数据从来没被删过。

SQLite 的笔记删了，ChromaDB 的向量数据还在。结果就是前端不显示这篇笔记了，但 RAG 搜索还能搜到它的内容。用户删了笔记之后再去提问，AI 照样引用"已删除"的笔记——体验很差。

## 为什么

`asyncio.create_task()` 创建了一个协程任务扔进事件循环，然后**立即返回**。它不保证任务什么时候执行、甚至不保证任务一定会执行完。

在这个场景里，`delete_note` 是同步方法，调用方 `notes.py` 的 FastAPI handler 是这样的：

```python
# notes.py (旧)
@router.delete("/{note_id}")
async def delete_note(note_id: str):
    deleted = note_service.delete_note(note_id)   # 调同步方法
    ...
```

FastAPI handler 本身在事件循环里运行，但 `delete_note` 是同步的，所以它在线程池里执行。`asyncio.create_task` 在这个上下文里的行为高度不确定：

- 如果事件循环刚好空闲，task 可能跑完
- 如果 HTTP 响应已经返回、事件循环进入下一轮，task 可能被取消
- task 里的异常也捕获不到，因为没人 await 它

说白了这就是"射后不管"（fire-and-forget），但 Python 的 asyncio 不像 Node.js 那样对 fire-and-forget 友好。Python 的协程你不 await，它就只是个悬空的 coroutine object。

## 修法

最直接的做法：**让 delete_note 也变成 async，好好 await。**

```python
# note_service.py (新)
async def delete_note(self, note_id: str) -> bool:   # ← 加 async
    session = db_manager.get_session()
    try:
        note = session.query(Note).filter(Note.id == note_id).first()
        if not note:
            return False

        if note.embedding_id:
            try:
                await vector_store.delete_by_note(note_id)   # ← 加 await
            except Exception as e:
                print(f"[WARN] ChromaDB 删除失败: {e}")

        session.delete(note)
        session.commit()
        return True
```

调用方也改：

```python
# notes.py (新)
@router.delete("/{note_id}")
async def delete_note(note_id: str):
    deleted = await note_service.delete_note(note_id)   # ← 加 await
    ...
```

就两行改动。但是——

## async 传染

这就是踩的第二个坑。方法签名加个 `async` 看起来简单，但它会**沿着调用链往上传染**。

调用 `delete_note` 的 `notes.py` handler 本来就可以 `await`（FastAPI handler 天然是 async），所以这一层没影响。但如果往上还有调用方是同步的，也得改。一层一层往上蹿。

在 Python 里，一旦你在调用链的某处引入了 async，它会强迫整个调用链都变成 async。你想在中间截断（比如 `asyncio.run()` 包一层同步），不是不行，但那又是另一种隐患（嵌套事件循环）。

这个项目的调用链比较浅（API route → service → DB），还好改。如果是一个深层嵌套的复杂系统，改一个底层方法就够受了。

## 什么时候 create_task 是合适的？

不是说 `create_task` 不能用。在 async 函数里，当你需要并行执行多个不相关的协程时，它是正确的工具：

```python
async def fetch_all(urls):
    tasks = [asyncio.create_task(fetch(url)) for url in urls]
    results = await asyncio.gather(*tasks)  # 等全部跑完
    return results
```

但前提是你在 async 上下文里，并且最终会 await 或者 gather 那些 task。

在同步方法里调 `create_task`，等于是把任务扔进一个你控制不了的事件循环，然后假装什么都没发生。这是 Bug 的温床。

## 总结

**Python 的 async 没有"真正的" fire-and-forget。你创建了一个协程，就得有人 await 它——要么你自己，要么 gather，要么扔给一个生命周期可控的后台任务队列。别假装它不存在。**
