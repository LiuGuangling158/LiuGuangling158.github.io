# Agent 系统开发踩坑日志

> 项目：LearnLoop-AI，一个 Multi-Agent 学习助手。  
---

## 1. `asyncio.create_task()` 的"射后不管"

**场景**：`note_service.py` 里 `delete_note()` 方法需要同步删 SQLite 和 ChromaDB 的数据。ChromaDB 操作是 async 的，图省事用了 `asyncio.create_task()`。

**现象**：删笔记后偶尔 ChromaDB 里的向量块还在。重新搜还能搜到已删的内容。

**原因**：`create_task()` 把协程丢给事件循环就跑，不保证它执行完。方法 return 了，task 才刚开始跑。ChromaDB 的删除操作就这样悄悄丢了。

**修法**：老老实实把 `delete_note` 改成 `async def`，里面 `await vector_store.delete_by_note()`。然后一路往上改——`notes.py` 的 handler 也得 `await`。改一个方法签名，传染到整个调用链。

> 教训：async 不是"加个关键字就快了"，是"你承诺等它跑完"。不想等就别用 async。

## 2. ChromaDB 距离值 ≠ 相关度，前端永远显示 0.00

**场景**：RAG 检索返回的 `score` 字段，前端显示全是 0.00。

**原因**：ChromaDB 返回的是 L2 距离（越小越相关），比如 0.83 表示距离 0.83。前端直接当"分数"展示，0.83 四舍五入显示成 0.00。而且前后端字段名不一致——后端返回 `score`，前端读 `relevance_score`。

**修法**：两步。第一，加个转换函数 `1.0 / (1.0 + abs(distance))`，把距离映射到 0-1 相关度。第二，字段统一为 `score`，在 `_enrich_sources()` 里按 title 匹配注入。

> 教训：向量数据库的"距离"和产品的"相关度"是两个概念。转换逻辑放在离数据源最近的地方。

## 3. `api_get` 把 Python `None` 当字符串 `"None"` 传

**场景**：笔记列表页有个"来源"筛选下拉框——全部/AI生成/上传的。选"全部"时传 `source_type=None`，以为后端收到的是空值。结果一条都筛不出来。

**原因**：`requests.get(params={"source_type": None})` → URL 变成 `?source_type=None`。FastAPI 收到的是字符串 `"None"` 而不是 Python `None`。后端 SQL 查 `WHERE source_type = 'None'`，没有匹配的行。

**修法**：`api_get` 里加一行 `params = {k: v for k, v in params.items() if v is not None}`。选"全部"时不传这个参数，后端默认返回所有。

> 教训：Python 的 `None` 和 HTTP 的"不传参数"是两回事。封装函数时顺手过滤 `None`，别靠直觉。

## 4. SQLAlchemy `expire_on_commit=False` 省了多少事

**场景**：笔记入库后立即读它的属性（id、title 等），报 `DetachedInstanceError`。

**原因**：`sessionmaker` 默认 `expire_on_commit=True`。commit 后所有已加载对象被标记为过期，再访问属性就抛异常。

**修法**：`sessionmaker(bind=engine, expire_on_commit=False)`。commit 后对象属性仍然可访问。

> 教训：SQLAlchemy 的默认值不是给 API 服务设计的——commit 后还要返回数据给前端的情况太常见了。这个配置该进项目模板。

## 5. Streamlit 的 `text_input` + `value=` 双绑问题

**场景**：搜索框的值从 `st.session_state` 读，又写回 `st.session_state`。点击重置按钮清空 session_state 后 `st.rerun()`，框里的字没变。

**原因**：`st.text_input(value=st.session_state.xxx)` + 紧接着 `st.session_state.xxx = return_value`，这在 Streamlit 里叫"双绑"。Streamlit 的 widget 自己维护了一个内部状态，和 session_state 不同步。重置时改了 session_state，但 widget 自己的状态没清。

**修法**：直接用 `key` 参数：`st.text_input(key="my_key")`。Streamlit 自动用这个 key 读写 session_state，不需要手动赋值。`selectbox` 同理，用 `key` 替代 `index` 参数。

> 教训：Streamlit 的 widget 要么全部交给 session_state 管理（用 `key`），要么全部手动（用 `value=` + `on_change`），不要混用。混用的结果就是"看着能跑，但总在某个边界情况崩"。

## 6. SM-2 状态表建好了但从来没人往里写

**场景**：`sm2_states` 表、SM-2 算法、schedule API 都写好了，三个独立存在。但打开 `/schedule/daily`，永远返回空列表。调试半天发现表里一条记录都没有。

**原因**：没有 Service 层。算法在 Agent 里，数据在 Model 里，API 在 Route 里，三层各写各的。创建状态的逻辑不知道该放哪，就没人写。

**修法**：新建 `sm2_service.py`，负责"从 API 层接收调用 → 查数据库 → 调 SM-2 算法 → 写回数据库"。然后让笔记生成的 API（`notes.py`）和做题提交的 API（`quiz.py`）在合适时机调用 `sm2_service.get_or_create_state(kp)`。

> 教训：当一个表没人往里写的时候，不是你忘了——是缺少一个"这件事归我管"的模块。

## 7. 前两次复习间隔写死的原因

**现象**：同样评 4 分，第一次复习间隔是 1 天，第二次是 6 天，第三次之后才开始 `interval × EF` 指数增长。一开始觉得是 bug。

**原因**：不是 bug，是 SM-2 的设计。一个全新的知识点，不能因为你一次就答对了就推到 10 天后。系统没有足够的证据判断你是"真的会了"还是"刚好蒙对"。用两次写死的短间隔（1 天、6 天）建立信心，之后再让 EF 接手。

> 教训：读算法不能只看公式，要看论文里为什么这么设计。那个"看起来像 bug 的 hardcode"可能是整个算法的核心防御。
