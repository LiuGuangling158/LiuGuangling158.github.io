# 把 45% 的 LLM 推理"省"掉——三级语义缓存架构

> **系列**：云平台智能客服系统技术深度解析 
> **关键词**：语义缓存、Milvus、Redis、余弦距离、LLM 成本优化


---

## 一、LLM 应用的成本真相

云平台智能客服上线后，我们统计到一个令人惊讶的数据：

> **45% 的用户问题在语义上与之前的问题高度重复。**

"C7.4xlarge 多少钱？" "C7.4xlarge 价格是多少？" "C7的4xlarge型号怎么收费？"——三个完全相同的语义，三次完整的 LLM 推理。

每次 ReAct Agent 推理消耗数千 Token，按 Qwen API 计费，日积月累的成本不容小觑。传统的 Redis key-value 缓存只能命中**完全一致**的文本，面对"语义相同但表述不同"的问题无能为力。

**语义缓存**正是为解决这一问题而生。

---

## 二、三级缓存架构

```
用户问题
   │
   ▼
┌────────────────────────────────────┐
│  L1: 精确匹配缓存 (Redis)           │
│  MD5(问题文本) → 答案               │
│  命中率 ~15% | 延迟 <1ms            │
└──────────────┬─────────────────────┘
               │ 未命中
               ▼
┌────────────────────────────────────┐
│  L2: 语义匹配缓存 (Milvus)          │
│  余弦距离 < 0.08 → 答案             │
│  命中率 ~30% | 延迟 ~5ms            │
└──────────────┬─────────────────────┘
               │ 未命中
               ▼
┌────────────────────────────────────┐
│  L3: 大模型推理                     │
│  Agent 完整执行 → 生成答案 + 回写缓存 │
│  命中率 ~55% | 延迟 2-30s           │
└────────────────────────────────────┘
```

**最终效果**：45% 的请求由缓存拦截。如果日均有 10000 次查询，4500 次不触发 LLM 推理，直接节省对应比例的成本。

## 三、L1：Redis 精确匹配——最快路径

```python
class ExactMatchCache:
    def _make_key(self, question: str, scope: str = "user") -> str:
        # 规范化后 MD5
        normalized = self._normalize(question)
        content_hash = hashlib.md5(normalized.encode()).hexdigest()
        return f"cache:exact:{scope}:{content_hash}"

    def _normalize(self, text: str) -> str:
        text = " ".join(text.split())              # 去多余空白
        text = text.replace("？", "?").replace("！", "!")  # 统一标点
        return text.lower()

    async def get(self, question: str) -> dict | None:
        # 先查 public → 再查 user
        public_key = self._make_key(question, "public")
        cached = await self.redis.get(public_key)
        if cached:
            return {**json.loads(cached), "cache_hit": "exact_public"}

        user_key = self._make_key(question, "user")
        cached = await self.redis.get(user_key)
        if cached:
            return {**json.loads(cached), "cache_hit": "exact_user"}

        return None
```

L1 用 Redis 而非 Milvus——精确匹配不需要语义能力，<1ms 的延迟才是核心价值。

## 四、L2：语义匹配——缓存的核心创新

L2 是整个缓存系统的大脑。核心参数是**余弦距离阈值**。

**余弦距离 = 1 - 余弦相似度**

```
距离 0.00 → 完全相同
距离 0.05 → 几乎相同（"多少钱" vs "什么价格"）
距离 0.08 → 高度相似 ← 我们的阈值
距离 0.15 → 中等相似（不同实例型号）
距离 0.30 → 弱相似（不同话题）
```

**0.08 是怎么定的？**——来自实验：

| 阈值 | 命中率 | 准确率 | 综合评估 |
|------|--------|--------|---------|
| 0.03 | 8% | 99.2% | 太保守 |
| 0.05 | 18% | 97.5% | 偏保守 |
| **0.08** | **32%** | **95.1%** | ✅ 最佳平衡 |
| 0.10 | 40% | 91.3% | 准确率已下降 |
| 0.15 | 52% | 82.7% | 太多误命中 |

0.08 在召回率（32%）和准确率（95%）之间取得最佳平衡。

## 核心要点

1. **45% 的缓存拦截率 = 近一半的推理成本节省**；
2. **L1 用 Redis（<1ms）+ L2 用 Milvus（~5ms）**——不同延迟要求对应不同存储；
3. **余弦距离 0.08 来自实验而非拍脑袋**——95% 准确率 + 32% 命中率的最佳平衡点。
