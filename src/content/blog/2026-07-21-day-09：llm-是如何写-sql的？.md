# LLM 是如何"写 SQL"的？

> **系列**：云平台智能客服系统技术深度解析
> **关键词**：Neo4j、Cypher、Text2Cypher、LLM 查询生成、安全校验、图数据库

---

## 一、Text2Cypher 的挑战

让 LLM 将自然语言转化为 Neo4j Cypher 查询，本质上等同于让它"写 SQL"。这比 Text2SQL 更难——Cypher 的图遍历语法对 LLM 来说更陌生。

**LLM 生成的 Cypher 可能出错的地方**：
- 编造不存在的节点标签（`MATCH (n:Server) → 实际是 InstanceType`）
- 关系类型拼写错误（`BELONGS_TO` → `BELONG_TO`）
- 语法错误（少括号、用错函数、Cypher 方言不兼容）
- 假阴性——查询执行成功但返回空，实际数据存在

## 二、System Prompt 的 Schema 注入

Text2Cypher 的质量上限由 Prompt 决定。关键是——**在图 Schema 中告诉 LLM 一切**：

```python
TC2C_SYSTEM_PROMPT = """你是 Neo4j Cypher 查询专家。

## 图数据库 Schema

### 节点及属性
1. InstanceType — 云服务器实例
   属性: name(STRING), cpu(INTEGER), memory(INTEGER),
         network_bandwidth(STRING), gpu(STRING|null),
         price_per_hour(FLOAT), price_per_month(FLOAT)

2. ProductSeries — 产品系列
   属性: name(STRING), description(STRING)

3. ProductFamily — 产品大类
   属性: name(STRING)

4. Region — 地域
   属性: name(STRING), code(STRING)

### 关系
- (InstanceType)-[:BELONGS_TO_SERIES]->(ProductSeries)
- (ProductSeries)-[:BELONGS_TO_FAMILY]->(ProductFamily)
- (InstanceType)-[:AVAILABLE_IN]->(Region)
- (InstanceType)-[:UPGRADE_TO {cost_ratio: FLOAT}]->(InstanceType)

### 约束规则
1. 只能生成 SELECT 查询，严禁 CREATE/DELETE/DROP/SET
2. 所有字符串比较使用 =，不用 CONTAINS（除非明确要求模糊搜索）
3. 默认 LIMIT 20
4. nullable 字段用 WHERE xxx IS NOT NULL
"""
```

**核心原则**：把 Schema 写得像 API 文档一样详细——节点有哪些、属性是什么类型、有哪些关系、关系有什么属性。LLM 不知道的就是不存在的。

## 三、Cypher 安全校验（多层防线）

LLM 输出的 Cypher 必须经过安全校验，绝不能直接执行：

```python
class CypherValidator:
    def validate(self, cypher: str) -> dict:
        issues = []

        # 第一层：危险操作检测
        DANGEROUS = ["CREATE", "DELETE", "DROP", "SET", "REMOVE", "MERGE", "DETACH"]
        for kw in DANGEROUS:
            if kw in cypher.upper():
                issues.append({"severity": "critical",
                              "message": f"包含危险操作: {kw}"})

        # 第二层：语法完整性
        if not cypher.strip().upper().startswith("MATCH"):
            issues.append({"severity": "warning",
                          "message": "不以 MATCH 开头"})
        if "RETURN" not in cypher.upper():
            issues.append({"severity": "error",
                          "message": "缺少 RETURN 子句"})

        # 第三层：括号匹配
        if cypher.count("(") != cypher.count(")"):
            issues.append({"severity": "error",
                          "message": "括号不匹配"})

        # 第四层：Schema 存在性
        known_labels = ["InstanceType", "ProductSeries", "ProductFamily", "Region"]
        if not any(label in cypher for label in known_labels):
            issues.append({"severity": "warning",
                          "message": "未引用任何已知节点标签"})

        return {
            "is_valid": not any(i["severity"] == "critical" for i in issues),
            "issues": issues
        }
```

四层校验从语法到语义逐级深入——危险操作 > 语法完整性 > 括号匹配 > Schema 存在性。任何"critical"级别的问题都直接拒绝执行。

## 四、当 Layer 1 失败时……

LLM 生成的 Cypher 可能有 ~15% 的概率出错（语法错误、Schema 幻觉等）。这些失败不会返回给用户——它们触发降级逻辑，进入 Layer 2（预定义模板）或 Layer 3（关键词全文检索）。

这是在 Day 10 中要展开的核心话题。

---

## 核心要点

1. **Text2Cypher 的 Prompt 决定质量上限**——投入精力写好 Schema 描述，远比事后修复错误更高效；
2. **绝不直接执行 LLM 生成的查询**——危险操作检测 + 语法完整性 + Schema 验证，多层校验缺一不可；
3. **预计 15% 的失败率是正常的**——这是为什么需要三层容错的根本原因。
